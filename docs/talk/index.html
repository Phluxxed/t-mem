Act 1 The Setup

Your agent has amnesia.

You debug the same Prisma schema bug today. You'll debug it again next Tuesday, and the Tuesday after that.

Most LLMs are stateless. Most agents are LLMs in a loop. So most agents have no systematic way to learn from execution.

"Agents have amnesia because most LLMs are stateless. An agent that struggles with a particular API authentication flow today will struggle with the same flow tomorrow unless its prompts are manually updated." — Paper §1, Introduction
Slow down here. Get a show of hands: "who's hit the same bug twice this week with their AI tool?" Most hands go up. That's the felt problem. Now name it: this is what the paper calls "agent amnesia." Set up the real cost: it's not just annoying — every time you re-discover, you burn tokens, time, and trust. The paper's framing: stateful systems learn, stateless systems suffer. We need a way to give stateless agents memory of their *experience* — not just facts, but lessons.

Act 1 The three tip types — for real

What a "tip" actually looks like.

Tips are structured, not free-form notes. Each has a category, a trigger, steps, a priority — the kind of record an agent can act on. These are real tips from a Claude Code session corpus I extracted with this pipeline.

Strategy High memory-agent session
Require assertions beyond same-session trigger events
whenMemory agent is about to write an observation after a session resume or continuation event.
doDiff proposed observation content against the most recent record. Only proceed if at least one new fact, decision, or state change is present.
Recovery High test-runner session
Bypass token-filtering proxy before concluding suite is empty
whenTest runner returns "No tests collected" via a token-filtering or output-processing wrapper.
doRe-run with a raw passthrough command before assuming there are no tests.

The full schema also carries: purpose, negative_example, source_session_id, task_context, an embedding, and a priority that retrieval uses for ranking. We'll come back to the full record when we get to storage.

These are real tips from my own corpus — 1,246 of them, extracted across 109 sessions. The first one is delicious because it's a memory agent that learned a rule about *when not to write a memory*. The agent literally taught itself something about its own job. The second one is a recovery tip with concrete corrective steps — note the specific trigger condition. Both have a category, a priority, a when, a do. That structure is what makes them retrievable and actionable. Free-form notes can't be filtered, ranked, or matched. Structured tips can. We're going to spend the rest of the talk understanding how the paper extracts, stores, and retrieves things in this shape.

Act 2 Paper §2.1 · Learning Requirements

Five things any honest learning system needs.

Most memory systems quietly skip several of these. The paper argues you need all five, or you end up storing facts instead of lessons.

1 · Strategy from clean wins

When an agent succeeds without errors, what made the run clean? Capture that, not just the outcome.

2 · Recovery from failures

When an agent stumbles and recovers, the recovery sequence is the lesson — not the failure or the eventual success alone.

3 · Optimization from inefficient wins

Succeeding suboptimally still teaches: which alternative would have been faster, cheaper, fewer calls?

4 · Step-level attribution

If failure happens at step 15, the cause may be at step 3. The system must trace the chain.

5 · Semantic reasoning analysis · provenance

Recognise why an agent thought what it thought (analytical / planning / validation / reflection patterns), and keep links from each tip back to the source trajectory so the learning itself can be debugged.

Walk these. Slow down on #2 — recovery is the requirement most existing memory systems get wrong, because they only record outcomes, not corrective sequences. #4 is the conceptually hardest: the failing step is rarely the buggy step. #5 is the one that lets you audit the system later — "why did the agent get this guidance?" needs to be answerable. If you can't trace a tip back to a trajectory, you can't tell good guidance from bad. Q for the room: "which of these does your existing setup get wrong?"
Act 2 Paper §2.3 · The status quo

Why the obvious fixes don't work.

Rule-based playbooks

Brittle. Can't adapt.

Hand-crafted rules require humans to anticipate patterns. The agent can't extend them from observed runs.

Prompt engineering

Manual feedback loop.

Improves only when a human notices a pattern and edits the prompt. No mechanism for systematic capture.

Generic memory (Mem0, Letta)

Stores facts, not lessons.

Treats all memories uniformly. No causal analysis. No category structure. No provenance back to trajectories.

Reinforcement learning

Opaque. Data-hungry.

Optimises for reward but won't separate strategy / recovery / optimization. Black-box improvements you can't audit.

Each of these solves part of the problem. None of them does structured, causal, categorised, traceable learning end-to-end. That's the gap the paper is filling.

This slide makes the paper's positioning clear. Each row points at a real system class people in the room have probably built or used. The honest framing: "all of these have value — the paper isn't saying they're wrong, it's saying none of them does the *whole* job." Generic memory is the closest comparable — Mem0 is well-engineered for what it does (storing user preferences, facts, entities) but it doesn't analyze *how* an agent succeeded or failed. The framework we're about to look at fills exactly that gap.

Phase 1 Paper §3.1.1 — §3.1.2 · Cognitive analysis & attribution

Read the agent's mind, then trace causes.

Trajectory Intelligence

An LLM classifies every assistant turn into one of four cognitive modes:

analytical

examining data, constraints

planning

formulating action sequences

validation

checking prerequisites, results

reflection

reconsidering, self-correcting

Plus pattern detection: validation, reflection, self-correction, error recognition, API discovery, efficiency awareness.

Decision Attribution

For every outcome, an LLM walks the chain backward to trace the cause:

Immediate  → what directly triggered it
Proximate  → recent decisions that enabled it
Root  → the underlying issue
Contributing  → factors that made it more likely

This is what the paper means by "semantic understanding": not pattern-matching strings in logs — actually reasoning about why the agent did what it did.

Two LLM passes. The first reads the trajectory and tags each turn with cognitive mode and pattern. The second walks the chain backward — "step 15 failed because of step 12 because of step 7 because of an unverified assumption at step 3." That cause-chain is what makes the eventual *recovery* tip actionable: it can name the *real* trigger, not the symptom. Important contrast: existing memory systems just log "task failed" or "task succeeded." This system logs "task failed because at step 3 the agent assumed payment was configured without checking, and the recovery was to call get_payment_methods() before retrying." That's a different ontology.

Optional Paper §3.1.4 · Granularity

Tip the whole task, or tip the parts?

Task-level tips

Tip the whole trajectory.

Captures end-to-end strategies (e.g. "verify all prerequisites before checkout"). High coverage, low transferability.

Subtask-level tips

Decompose first, then tip each part.

Authentication / data-retrieval / data-processing / task-completion subtasks recur across apps. Tips on those transfer across tasks.

A tip extracted from "Name the artist most recommended on Spotify" can transfer to "Move my go-to-sleep alarm to 20 minutes later" — they share the same authentication and paginated data-retrieval subtasks. — Paper §3.1.4

The paper uses both granularities together. Subtask tips give cross-task transfer; task tips capture the holistic strategy.

OPTIONAL slide. Skip if running long. The insight: most agent tasks decompose into common shapes — auth, fetch, process, report — and tips at *that* level transfer between apps that share those shapes. Spotify's auth and Phone's auth look different on the surface but the same underneath. A subtask-level tip about retrieving credentials applies to both. Task-level tips like "verify prerequisites" are cross-cutting at a different level.
Phase 2 Paper §3.2 · Storage & Management

Don't just dump tips in a database.

A thousand sessions produce thousands of redundant, overlapping, sometimes contradictory tips. Three steps clean them up before they go to disk.

1 · Generalize

Strip the specifics.

"Retrieve Spotify password for john.doe@…" → "Retrieve service account credentials"

Entity abstraction · action canonicalization · context removal.

2 · Cluster

Group near-duplicates.

Embed each generalised description; pairwise cosine similarity ≥ 0.85 → cluster.

Hierarchical agglomerative clustering. Tips from Spotify auth, Venmo login, Phone login all land together.

3 · Consolidate

LLM merges each cluster.

Dedupe identical tips. Resolve conflicts (success-derived beats failure-derived). Synthesize complementary tips into one.

Output: one canonical tip per cluster, with merged steps and provenance to all source trajectories.

Each stored tip carries two indexes: a vector embedding for semantic search, and structured metadata (category, priority, application, source trajectories) for filtering and ranking.

Three steps, all on one slide because they're a single conceptual story: prepare → group → merge. Generalisation is where most existing systems quietly cheat — they cluster on raw text, which means "get Spotify password for alice" and "get Spotify password for bob" look like different tips. Strip the entities, embeddings cluster correctly, you get a curated corpus instead of an exponentially-growing junk drawer. Conflict resolution: when two tips disagree, the system trusts the one derived from a *successful* trajectory over one derived from a failed one. That's a clever heuristic.
Phase 3 Paper §3.3 · Runtime Retrieval

How a tip finds its way back into a prompt.

Strategy A · Cosine similarity

Embed task → match → top-k.

Embed the user's task description, run cosine similarity against every stored tip embedding, keep candidates with similarity τ ≥ 0.6 and take the top k=5. One vector lookup, no LLM call — fast enough to run on every prompt.

τ = 0.6 is the sweet spot. Tight enough to skip unrelated tasks; loose enough to match lexically-different phrasings of the same intent.

Strategy B · LLM-guided

Reason about context first.

An LLM detects the application domain and task category, then constructs a metadata-filtered query before the vector search runs. It can infer that "complete my pending checkout" implies payment tips even when the word "payment" never appears in the user's prompt — the kind of cross-vocabulary reasoning cosine similarity misses entirely.

Worth +7.2pp SGC over cosine alone — at the cost of one extra LLM call per task. Best on multi-domain workloads where vocabulary and intent diverge.

What the agent actually sees, prepended to its prompt:

[PRIORITY: HIGH] Recovery Tip:
  When a login attempt fails with "invalid credentials," verify
  you are using the correct app-specific password by re-calling
  supervisor.show_account_passwords() and filtering by the target
  app name.

Apply when: Authentication fails on any app after an initial login attempt.
Steps:
  1. Re-retrieve credentials from supervisor
  2. Filter for the specific app name (exact match)
  3. Retry login with the correct credentials
Two strategies, two tradeoffs. Cosine is the simple workhorse — embed the task, find similar tip embeddings, take the top few. LLM-guided is the upgrade: it can reason "this task involves checkout therefore payment-related tips matter even if 'payment' isn't in the description." More expensive, more accurate, especially on complex tasks. The injection format at the bottom is what the agent actually sees in its context — priority tag, category, condition, action, steps. It reads like guidance, not a database row. That formatting matters; the paper is explicit that scannable structure helps the model use the tip.

Act 4 The most insight-dense number

Why SGC > TGC matters more than the headline.

Memory's biggest impact isn't "can the agent do this task?" — it's "does it do it the same way every time?"

On test-normal, TGC went up 3.6pp. SGC went up 14.3pp — nearly 4× more. The memory system reduces behavioural variance: same scenario, three variants, all three pass instead of two.

That's the "I fixed this once and it worked, but the second time I tried, the agent did it differently and broke" pattern. Recovery and strategy tips encode prerequisite verification — so it stops being random.

This is the most underrated insight in the paper. SGC vs TGC is the difference between "can my agent do X?" and "can my agent do X reliably?" Anyone in the room who's used Claude Code or Cursor in a real job knows the second question is the one that actually matters — the agent that nails it 8 times out of 10 but fails the 9th in a weird way is more frustrating than one that succeeds 7 out of 10 consistently. The memory system reduces *behavioural variance* — it teaches the agent to verify the same prerequisites every time, which means scenario variants stop randomly failing. That's why SGC moves twice as much as TGC. Pause for discussion here — this is where the paper's thesis becomes felt rather than abstract.

Phase 2 in code

Cluster, then merge.

Pairwise cosine similarity union-finds tips into clusters. The LLM decides what to do with each cluster.

def find_clusters(store, threshold=0.88, tip_ids=None):
    """Find groups of tips with pairwise cosine similarity >= threshold."""
    rows = store.get_tips_with_embeddings(get_available_provider())
    embeddings = {row["id"]: row["embedding"] for row in rows}
    ids = list(embeddings.keys())

    uf = _UnionFind(ids)                     # disjoint-set
    for i in range(len(ids)):
        for j in range(i + 1, len(ids)):
            sim = _cosine_similarity(embeddings[ids[i]], embeddings[ids[j]])
            if sim >= threshold:
                uf.union(ids[i], ids[j])     # merge into the same cluster

    return [Cluster(tips=tips_in(group)) for group in uf.clusters() if len(group) >= 2]


def decide_merge(cluster, *, model="sonnet"):
    """LLM judges: are these the same tip? Synthesise into one canonical version."""
    raw = call_claude(build_merge_prompt(cluster), model=model)
    return _parse_merge_response(raw, source_tips=cluster.tips)

Two phases on purpose. Embedding similarity is cheap and finds candidates. The LLM is expensive but reasons about whether the candidates actually convey the same insight — and synthesises them into a single tip with merged provenance.

The pattern: cheap-first, expensive-second. Cosine similarity at 0.88 is a recall-oriented filter — it casts a wide net, lots of pairs get clustered together. The LLM then makes the precision call, "are these actually the same tip or just superficially similar?" This is a common pattern in modern ML pipelines — fast vector search to narrow the candidate space, slower reasoning model to make the actual decision. Worth pointing out: the threshold (0.88) is empirical; the paper found ~0.85 worked for AppWorld; I tuned it slightly tighter for my Claude Code corpus because the embeddings are richer.
Phase 3 in code The Claude Code integration

How a tip lands inside your next prompt.

The retriever — runs on every prompt:

def retrieve_tips(query, store, *, threshold=0.6,
                  top_k=5, session_id=None):
    """Cosine retrieval with abstracted query."""
    # 1. strip entities so the query matches generalised tips
    abstracted = abstract_query(query)
    q = embed_text(abstracted)

    stored = store.get_tips_with_embeddings(q.provider)
    seen = store.get_injected_tip_ids(session_id) or set()

    scored = []
    for tip in stored:
        if tip["id"] in seen:                # dedup per session
            continue
        s = _cosine_similarity(q.vector, tip["embedding"])
        if s >= threshold:
            scored.append((s, tip))

    scored.sort(reverse=True)
    return [store.get_tip(t["id"]) for _, t in scored[:top_k]]

Act 5 Honest tradeoffs

Where I adapted the paper.

Not every difference is a cut. Some paper machinery became a simpler implementation boundary.

Kept · Task + subtask granularity

The repo does both levels the paper recommends: segment into transferable subtasks, then run one whole-session pass for holistic strategies that span the full trajectory.

What I skip is the paper's separate domain-specific + generic tip records; mine bias toward one transferable/generalised record.

Cut · LLM-guided retrieval

Worth +7.2pp SGC over cosine alone in the paper. I ship cosine plus a small Haiku query-abstraction call; the cut is the paper's fuller LLM-guided domain/metadata selector.

Trade-off: more retrieval-time reasoning vs. latency on every interaction. Punt until the corpus is big enough to need it.

Adapted · Generalization at the boundaries

Generalization is explicit: the segmenter writes generalized_description with entity abstraction, action normalization, and context removal; retrieval abstracts the user prompt before embedding.

What changed from the paper is placement: not a separate storage-stage pass, but built into extraction and retrieval.

Next · "Design principle" tip category

A fourth category for meta-cognitive lessons, closer to ReasoningBank: "when calling an external API in a loop, think rate limits"; "always run pip-show before importing." Different abstraction level.

Would need its own extraction prompt tuned for reasoning patterns rather than concrete actions.

Honest about the gap, not embarrassed by it. My implementation keeps the core loop, task/subtask granularity, and privacy-preserving generalization. The big remaining paper feature is full LLM-guided retrieval — richer selection logic, paid for at prompt time.

Clarify terminology here. "Dual-level" can mean two different things. Task-level plus subtask-level granularity is implemented. The domain-specific/generic distinction is handled more pragmatically: prompts push tips toward transferable/generalised language rather than always writing two separate records. Generalization is not a nice-to-have here — it is a privacy boundary, because work session details should not leak into reusable memory or embedding queries. The real remaining paper gap is full LLM-guided retrieval: richer domain/metadata reasoning before selection.
